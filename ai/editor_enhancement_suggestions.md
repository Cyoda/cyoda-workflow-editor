**1. Executive summary**
The repo already has a real editor shell, not just a viewer: `@cyoda/workflow-react` renders a React Flow canvas, inspector, undo/redo stack, drag-connect modal, delete-state modal, save-flow helpers, and viewport persistence. The canonical architecture is mostly aligned with your constraints: Cyoda workflow JSON is the source of truth, graph/layout are projections, export serialization is deterministic, and editor-only state already lives outside export JSON in `meta`.

What is missing is the full workflow-editing surface. State add is absent; state edit is basically rename/delete only; transition edit is partial; criteria editing is effectively missing; processor editing is partial; manual layout persistence is stubbed but unused; Monaco/canvas sync exists only in the demo app, not in the packaged editor; and undo inverses are still coarse for many edits. I did not implement or run changes; this is code inspection plus the existing test surface.

Every patch op has some inverse through [invert.ts](/Users/patrick/dev/cyoda-workflow-editor/packages/workflow-core/src/patch/invert.ts), but many are coarse `replaceSession` fallbacks. Exact inverses exist for `addWorkflow`, `updateWorkflowMeta`, `setInitialState`, `setWorkflowCriterion`, `addState`, `updateTransition`, `updateProcessor`, `setCriterion`, `setImportMode`, `setEntity`, and `setEdgeAnchors`; `removeWorkflow`, `renameWorkflow`, `removeState`, `renameState`, `addTransition`, `removeTransition`, `reorderTransition`, `addProcessor`, `removeProcessor`, `reorderProcessor`, and `replaceSession` all invert coarsely. Validation surfaces to React as toolbar counts, node warning/error highlighting through graph annotations, inspector issue panels, and save-button disablement. Patches are granular enough for many transition/processor field edits, but JSON editing is whole-session `replaceSession`, and cascading state/workflow operations are still coarse.

Graph projection: `projectToGraph` in [packages/workflow-graph/src/project/project.ts](/Users/patrick/dev/cyoda-workflow-editor/packages/workflow-graph/src/project/project.ts) maps each workflow to state nodes plus a synthetic start-marker node/edge, and each transition to a `TransitionEdge` with summaries for criteria/processors/execution. State node IDs are the synthetic state UUIDs from `meta.ids.states`; transition edge IDs are the synthetic transition UUIDs from `meta.ids.transitions`; workflow synthetic IDs exist only in metadata, not as graph nodes; processor and criterion UUIDs exist only in metadata, not as graph nodes. Self-transitions are `isSelf: true`; parallel transitions are grouped by `(sourceId,targetId)` and carry `parallelIndex` and `parallelGroupSize`; terminal states are states with zero outgoing transitions; the initial-state marker is a `startMarker` node with ID `startmarker:${workflow}` plus a non-interactive edge to the initial state. Disabled/manual are booleans on `TransitionEdge` and drive styling in viewer/react. The graph model has no positions and cleanly separates layout from domain JSON. It does support criteria/processors/execution badges through edge summaries, and validation annotations through `GraphAnnotation`, but not separate criterion/processor nodes.

5. Layout system: automatic layout lives in `@cyoda/workflow-layout`, primarily [adapter.ts](/Users/patrick/dev/cyoda-workflow-editor/packages/workflow-layout/src/adapter.ts). The engine is `elkjs/lib/elk.bundled.js`. The actual implementation is async main-thread only; there is no separate worker adapter today, even though README/package descriptions mention worker support. Presets are implemented as `websiteCompact`, `configuratorReadable`, and `opsAudit`. Layout results are passed to the viewer as a `layout` prop and computed internally inside `WorkflowEditor`’s canvas. Results are intended to be deterministic and are tested for stability, but they still depend on ELK version/ordering behavior. There is no lock-layout API; the only manual-position API is `LayoutOptions.pinned`, which already supports mixing fixed positions with auto-layout for other nodes. The cleanest storage that does not contaminate exported Cyoda JSON is editor-only `meta.workflowUi` or a host-managed companion payload with the same shape; `serializeImportPayload` and `serializeExportPayload` already exclude it.

6. React Flow integration: `reactflow` is a peer dependency `^11`, with `11.11.4` in dev/demo packages. Nodes are draggable when not read-only, and that is not intentionally disabled, but the editor does not persist or reconcile node movement. `onConnect` is wired; `onMoveEnd` is wired for viewport persistence; `onNodesChange`, `onEdgesChange`, `onNodeDragStop`, keyboard delete, copy/paste, and multi-select state handling are not wired. Node and edge renderers are custom. Selection propagates from React Flow via node/edge click handlers into `useEditorStore`, then into the inspector. Canvas and inspector share state/transition/processor selection reliably enough; workflow and criterion selection do not. Viewport pan/zoom is persisted per workflow and orientation in `meta.workflowUi[workflow].viewports`. For a full editor, add `onNodeDragStop`, `onNodesChange`, `onSelectionChange`, node/edge delete hooks, connect-start/end affordances, reconnect/retarget hooks, and keyboard command handling.

7. State editing UX: based on the domain model in [types/workflow.ts](/Users/patrick/dev/cyoda-workflow-editor/packages/workflow-core/src/types/workflow.ts), a state has no editable fields beyond its name and its outgoing transition list. The current state editor should therefore expose name, derived status badges, transition count, delete, and initial-state control. Renaming a state should update the `states` map key, `initialState`, and every transition `next` pointer; current code already does that in [patch/apply.ts](/Users/patrick/dev/cyoda-workflow-editor/packages/workflow-core/src/patch/apply.ts), but it does not preserve the state UUID because state IDs are reused by `(workflow,stateCode)`. Validation needs `NAME_REGEX`, non-empty, trimmed, and collision checks. Today a rename to an existing state name would silently overwrite that state, which is a serious risk. Terminal states should stay visually marked and not get special editable fields. Initial state should be changed through a state chooser or workflow dropdown, not a free-text field.

8. Add-state UX: no current implementation exists. Recommended placement is toolbar plus keyboard shortcut first, with canvas context-menu placement when manual layout lands. Default names should be deterministic and collision-free, such as `state1`, `state2`, and so on. New states should be allowed as terminal/unreachable initially; unreachable should remain a warning, not an error, to support incremental editing. If creation happens from a canvas click, use that point as the initial manual layout position once layout persistence exists. The patch should be `addState`; undo should be `removeState`; if position is created too, wrap both in a single transaction with a layout-meta change.

9. Transition editing UX: transitions are currently selected from the canvas by edge click and edited in the inspector. The editor should own `name`, `next`, `manual`, `disabled`, `criterion`, and `processors`; right now criterion is the missing part. Source state should not be mutated through a generic `updateTransition`; if you want source moves, add a dedicated `moveTransition` patch or equivalent transaction. Target should be changeable both from inspector and by drag-end retargeting. Duplicate transition names are currently only validated within a source state, not globally, and that is the right uniqueness scope for this model. Self-transitions are already visually supported. Parallel edges are already modeled, but their selection/hit area needs hardening because labels are overlay DOM and path-click selection is the only explicit wiring.

10. Drag-connect UX: there is already a drag-connect modal in [DragConnectModal.tsx](/Users/patrick/dev/cyoda-workflow-editor/packages/workflow-react/src/modals/DragConnectModal.tsx). A same-workflow state-to-state drag opens the modal; no transition is created until the user enters a valid unique name and confirms; cancel creates nothing. The current modal does not suggest a default name. Current defaults are `manual: false` and `disabled: false`. Self-transitions and multiple source-target parallels are both allowed by the current code. The patch family is `addTransition`; today undo falls back to `replaceSession`, so exact undo for created transitions is still incomplete.

11. Criteria editor UX: current form support is effectively absent; criteria only show up as edge-summary badges and raw JSON preview. The full editor should support both structured forms and a raw JSON escape hatch. Simple criteria need `jsonPath`, `operation`, `value`; group criteria need recursive add/reorder/delete of nested conditions; function criteria need function name/config/local criterion; lifecycle criteria need `field`, `operation`, `value`; array criteria need `jsonPath`, `operation`, and editable string lists. Temporary invalid drafts should live in local form state; patches should only commit canonical JSON once the structure is syntactically valid enough. Nested criteria editing needs dedicated tests for recursion, reordering, undo, and serialization.

12. Processor editor UX: the domain supports `externalized` and `scheduled`. Externalized processors need form support for `name`, `executionMode`, `attachEntity`, `calculationNodesTags`, `responseTimeoutMs`, `retryPolicy`, `context`, `asyncResult`, and `crossoverToAsyncMs`. Scheduled processors need `name`, `delayMs`, `transition`, and optional `timeoutMs`. Current UI only fully supports an externalized subset; scheduled processors are largely unedited beyond name/reorder/delete. Processor order is currently button-based; drag reorder can come later. A raw JSON escape hatch is advisable. Processor names should remain unique per transition. `executionMode` should default to `ASYNC_NEW_TX` in UI and continue to be omitted in deterministic output when default. Schema failures, duplicate names, and broken scheduled targets should block save; `crossoverToAsyncMs` without `asyncResult` and processor-overload should stay warnings.

13. Manual layout UX: users can drag states today, but those positions are not stable or persisted. They should be able to drag freely, and dragged positions should survive later auto-layout by becoming pinned/manual positions. There is no lock-layout mode today; a reset button and an auto-layout button should come with persistence, and an “auto-layout unpositioned only” command can be built directly on the existing `pinned` API. Manual layout should not be exported inside Cyoda workflow JSON. The clean API is a host-controlled layout/ui-state payload loaded and saved separately from workflow JSON. If Monaco deletes states, stale layout entries must be cleaned during `replaceSession`. Layout should use semantic keys like `(workflowName,stateCode)`, not synthetic IDs.

14. JSON editor synchronization: Monaco sync exists, but only in the demo page [MonacoPlaygroundPage.tsx](/Users/patrick/dev/cyoda-workflow-editor/apps/docs-embed-demo/src/pages/MonacoPlaygroundPage.tsx), not inside `WorkflowEditor`. Canvas selection can reveal JSON there, and JSON cursor can select canvas objects there, through `attachCursorSelectionBridge` and `revealIdInEditor`. JSON edits reuse workflow/state IDs by name, transition IDs by source-state ordinal, and processor IDs by transition ordinal; criterion IDs are not reused at all. Manual layout survival across JSON edits is not implemented; current `replaceSession` keeps prior `workflowUi` wholesale, so future layout metadata would need cleanup/migration logic. Undo/redo is not shared between Monaco and canvas today. Invalid JSON and invalid-schema edits are isolated from canonical state; schema-valid semantic-error edits are currently allowed through.

15. Undo/redo and command history: undo/redo lives in [packages/workflow-react/src/state/store.ts](/Users/patrick/dev/cyoda-workflow-editor/packages/workflow-react/src/state/store.ts). Every UI dispatch records an inverse, but many inverses are coarse `replaceSession` snapshots rather than exact patch inverses. Form edits are committed on blur, checkbox toggle, or select change. Drag movement currently produces no undo because there is no persisted node-position edit. Rename-state is already one logical patch and should stay one undo step. There is no patch transaction abstraction; one should be introduced. Recommended shape: `PatchTransaction { summary: string; patches: DomainPatch[]; inverses: DomainPatch[]; selectionBefore?: Selection | null; selectionAfter?: Selection | null; }`.

16. Validation and user feedback: current semantic rules are in [validate/semantic.ts](/Users/patrick/dev/cyoda-workflow-editor/packages/workflow-core/src/validate/semantic.ts) and include duplicate workflow names, missing/unknown initial state, invalid names, unknown transition targets, duplicate transition names, duplicate processor names, scheduled target issues, disabled transition in active workflow, excessive fan-out, all-manual states, derived terminal states, unreachable states, inactive workflows, sync processors on auto-reachable transitions, function criterion issues, array non-string values, lifecycle field validity, and `NOT` group warnings. Save is blocked when there is any `error`; warnings and infos do not block. Unreachable states are detected. Missing targets are detected. Duplicate transition names are detected per source state. Invalid state names are detected. Invalid criteria and processor configs are partly schema-level and partly semantic. Canvas mapping is partial because many issues lack `targetId`; Monaco markers also fall back to root when they cannot resolve a target. A full editor should add rename collision validation, state-delete impact validation, layout-reference validation, better per-transition/per-criterion targeting, and inline inspector errors plus canvas badges plus Monaco markers.

17. Public API design: `@cyoda/workflow-react` currently exports `WorkflowEditor`, `useSaveFlow`, `SaveConfirmModal`, `ConflictBanner`, `diffSummary`, i18n context/helpers, and some layout/editor types. Public hooks are `useSaveFlow` and `useMessages`. `@cyoda/workflow-core` already exports the core editing types: `WorkflowEditorDocument`, `DomainPatch`, `WorkflowUiMeta`, `EditorViewport`, `ValidationIssue`, `WorkflowApi`, all criterion/processor/workflow types, and patch/parse/serialize helpers. New public APIs should be additive: controlled document editing (`onPatch` or transaction callback), controlled selection, controlled layout/ui metadata, explicit read-only flag separate from mode, custom validation hook/adapter, host persistence hooks, import-mode/save hooks, and i18n message overrides. Current exports are explicit rather than wildcard and are reasonably stable. Adding full editor features can stay non-breaking if it is additive; changing synthetic ID behavior or current `WorkflowEditorProps` semantics would be a breaking change.

18. Package boundary review: `workflow-core` should own canonical document types, patch families, inverse logic, validation, deterministic serialization, and editor-only metadata types. `workflow-graph` should stay the pure projection layer plus graph-originated edit translation. `workflow-layout` should own ELK routing, manual-pinned merge behavior, reset/auto-layout helpers, and eventually worker offload. `workflow-react` should own React Flow wiring, inspector UX, command history, keyboard shortcuts, and host-controlled editing APIs. `workflow-monaco` should own schema registration, marker mapping, JSON patch lifting, and shared-history integration points. `workflow-viewer` should stay read-only rendering plus shared theme tokens and visual conventions. Manual layout metadata type can stay in `workflow-core` because `WorkflowUiMeta` already exists there, but layout mutation helpers should not live in the viewer. React Flow-specific code is already isolated to `workflow-react` and should remain that way. Viewer code is not incorrectly depending on editor-only concepts today.

20. Accessibility and keyboard UX: keyboard-accessible today are toolbar buttons, workflow tabs, inspector form fields/buttons, save/confirm/conflict modals, and modal Escape-to-close. Node selection is not keyboard-accessible in the React Flow canvas, and edge selection is not either. State add and transition add cannot currently be completed without a mouse. Modals restore focus and set dialog semantics, but they are not fully focus-trapped. Validation counts are announced via toolbar pills; inspector issue text is not announced live. Shortcuts to add: add state, delete selected, edit selected, auto-layout, save, undo/redo, and probably cycle selection. Missing ARIA is mainly on canvas nodes, edges, drag handles, and future toolbar/context actions.

Persistence and backend integration: the save contract is `WorkflowApi` in [packages/workflow-core/src/types/api.ts](/Users/patrick/dev/cyoda-workflow-editor/packages/workflow-core/src/types/api.ts), plus `useSaveFlow` in [packages/workflow-react/src/save/useSaveFlow.ts](/Users/patrick/dev/cyoda-workflow-editor/packages/workflow-react/src/save/useSaveFlow.ts). Concurrency conflicts are handled with an opaque token and `WorkflowApiConflictError`. There is no backend endpoint or public callback for layout metadata today. A host callback API should be added for editor-ui metadata separately. Save should treat workflow JSON and layout metadata as separate payloads.

22. Performance and scalability: explicit budgets exist only for core parse/validate/serialize/patch in [README.md](/Users/patrick/dev/cyoda-workflow-editor/README.md) and [packages/workflow-core/tests/perf/bench.bench.ts](/Users/patrick/dev/cyoda-workflow-editor/packages/workflow-core/tests/perf/bench.bench.ts). Benchmarks are present for core and not for React Flow or ELK/editor shells. There is no measured editor behavior in repo for 10, 50, 100, or 500-state workflows. React Flow is probably fine for small/medium graphs, but the current editor recalculates full projection, validation, and ELK layout for every session change, so 100+ states are a risk and 500 is unverified. ELK is main-thread and async-only; worker support should be completed before large-graph editing is treated as a requirement. Expensive validation/layout should be debounced or offloaded. Avoidable architecture risks include full-layout reruns on every edit, criterion ID churn, and one important bug: `WorkflowEditor`’s derived graph memo ignores `meta.workflowUi`, so anchor-only edits can go stale.

23. Visual design: current conventions are mostly centralized in `workflow-viewer/theme`. Initial states use a start marker and green initial palette; terminal states use terminal role and double ring; manual transitions use purple styling and a manual badge; automated transitions use the neutral lane; disabled transitions are gray and dashed with a disabled badge; criteria use an amber badge/conditional lane; processors use blue badges/lanes; selection is a dark border/stroke emphasis. Validation error/warning styling currently exists only in the React Flow editor nodes, not the viewer. Viewer and editor share the same visual language and tokens, but the editor duplicates node/edge renderers in React Flow rather than literally reusing viewer components. Full editor work still needs visible edit handles, stronger drag-connect affordances, validation badges, better selected-edge hit areas, and likely inline labels/handles for parallel/self edges.

24. Compatibility and release impact: a full implementation likely changes `workflow-core`, `workflow-graph`, `workflow-layout`, `workflow-react`, and `workflow-monaco`; `workflow-viewer` only if you change shared visuals/tokens. Public types will almost certainly grow, and there is breaking-risk if you change synthetic ID semantics, `WorkflowEditorProps`, or mode behavior. Existing `workflow-react` and `workflow-monaco` consumers are the most likely to notice behavior changes; viewer-only consumers should remain safe if React Flow stays isolated. Public package changes require a changeset. Bundle-size risk is mainly in `workflow-react`; the viewer package should remain clean because it currently has no `reactflow` dependency. Private packages remain private: the root `cyoda-workflow-editor` package and `@cyoda/docs-embed-demo` are both marked `private`.

**4. Proposed technical design**
- Keep `WorkflowEditorDocument.session` as the only canonical workflow state; keep all layout, viewport, anchor, and collapse data in editor-only `meta.workflowUi`.
- Add a transaction layer above raw `DomainPatch` so one user action can produce multiple canonical and editor-ui changes with one undo step.
- Extend patch families conservatively: new patches should stay semantic and JSON-oriented, not React Flow-oriented.
- Keep graph projection free of layout and React Flow concerns; layout stays a separate package output.
- Persist manual layout by semantic path `(workflowName,stateCode)` and migrate it during rename/delete/replace operations.
- Treat Monaco edits as first-class commands into the same store, not a parallel side-channel.
- Add controlled UI-state props to `WorkflowEditor` rather than baking backend persistence assumptions into the package.
- Preserve deterministic serialization by continuing to serialize only known canonical fields through `outputWorkflow` and friends.

**6. Open questions that require product input rather than code inspection**
- Should unreachable states remain warnings, or should some flows elevate them to errors before save?
- Should workflow properties be edited from a dedicated workflow inspector target, or only through a workflow-metadata panel outside selection?
- Do you want transition source-state editing at all, or should moving a transition between states be intentionally unsupported?
- For criteria and processor config, should the UI allow temporarily invalid drafts, or must every intermediate commit stay canonical?
- Should manual layout be persisted by the host app as a separate artifact, or do you want the app to persist full editor documents including `meta`?
- What is the desired large-graph target: “comfortable at 100 states” or “must remain usable at 500 states”? That determines whether layout worker work is prerequisite or polish.
- Should state rename preserve the visual/editor identity of the node across rename, or is minting a new state synthetic ID acceptable as long as exported JSON stays correct?
- Should the editor expose one combined JSON+visual history, or may Monaco edits have a coarser history model than inspector/canvas edits?
- What keyboard shortcut set do you want standardized for add state, delete selection, edit selection, and auto-layout?
- Should the public `WorkflowEditor` API stay opinionated and bundled, or do you want lower-level controlled hooks/components for host apps with custom chrome?
